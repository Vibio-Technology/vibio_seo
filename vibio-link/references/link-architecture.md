# 内链架构操作手册

内链帮助用户和搜索引擎发现页面、理解页面关系、沿任务前进，并把站点已有的链接信号分配到重要内容。它是可控系统，但没有通用的点击深度、每千字链接数或 AI 引用倍数。

## 1. 目标

1. 希望参与搜索的页面没有孤儿。
2. 重要页面可从相关导航、hub 或正文路径到达。
3. 链接出现在用户真正需要下一步信息的位置。
4. 锚文本准确预告目标页面，不操纵、不误导。
5. 新页面发布、合并、迁移和删除后，链接图同步更新。
6. 内链改动能通过 crawl/render 和后续一方数据复盘。

## 2. 先建立真实链接图

合并以下来源：

- 渲染后 crawl 的页面与 `<a href>` 边。
- Sitemap 中希望被发现的 canonical URL。
- GSC 已知页面和内部链接报告（覆盖有限，不能单独当完整图）。
- CMS/内容库存与代码生成的数据源。
- Redirect、canonical、hreflang 和状态码。

动态导航、客户端组件和 CMS 页面不能只靠源码 grep。无法完整抓取时明确 coverage。

每条边至少记录：source、target、位置（正文/导航/页脚/面包屑/组件）、anchor、状态、是否模板化。

## 3. 站点拓扑

可以采用 hub -> task/page -> detail 的多层结构，但层级由真实产品、任务和用户旅程决定：

```text
核心商业/支柱页
  <-> 应用、比较、选型或主题页
       <-> 规格、案例、故障、问题页
```

规则：

- Hub 应链向真正属于它的成员，成员也应提供回到更宽任务/商业路径的链接。
- 真实相关时允许跨簇链接；不要为了 silo 拒绝用户需要的跳转。
- 导航和面包屑保障结构与可达性，正文链接提供具体上下文；两者都有效，但职责不同。
- 商业价值高不代表要从所有页面硬链；相关性和用户路径优先。

## 4. 锚文本

- 用自然语言说明目标页会提供什么。
- “点击这里/了解更多”在缺少上下文时不够清楚；若周围文本已明确，不必机械改写。
- 站内可以自然使用目标术语，但不要全站批量复制同一句精确匹配锚。
- 同一 anchor 指向多个页面不自动等于 cannibalization；只有当页面意图和搜索表现也冲突时才处理。
- 图片链接使用准确 alt/可访问名称帮助理解目标。

不维护为了比例而存在的“锚库”。变体应来自不同上下文，不是人为轮换。

## 5. 新页面发布协议

发布前在 brief 中确定：

1. 哪些现有页面的读者会自然需要新页。
2. 哪个 hub/导航需要加入新页。
3. 新页应链向哪些前置、后续和商业页面。
4. 每条链接放在哪里、为什么帮助用户。

发布当天：

- 从最相关的现有页面建立至少一条真实可抓取入链；通常选择 2-3 个不同上下文 donor，但不强制凑数。
- 更新所属 hub/列表/导航。
- 检查新页链出的目标均为最终 canonical URL，无 4xx 或无意义 redirect chain。
- 重新 crawl 或检查渲染结果，把动作写入 `.vibio/changelog.md` 与 links tracker。

没有入链的新页面应标为未完成发布，不宣称其价值“固定减半”。

## 6. Donor-Acceptor 机会

Donor 候选来自：

- 真实获得外链、organic traffic 或稳定曝光的相关页面。
- 同一买家任务/产品路径中的权威 hub。
- 已提到 acceptor 主题但没有提供下一步的页面。

Acceptor 候选是业务重要、内容合格、可索引，但相关发现/支持不足的页面。

每个建议都要说明 source -> target 的用户理由。不能因为 donor 流量高就塞固定数量链接。记录改动，并以 crawl/index、GSC 页面组趋势和合格 landing behavior 复盘。

## 7. 审计检测项

| 检测 | 证据 | 处理 |
|---|---|---|
| 孤儿页 | 内容库存/sitemap/GSC 有，渲染 crawl 无入链 | 确认页面是否应保留；需要则从相关位置加链 |
| 断链 | Internal edge 指向 4xx/5xx | 修到正确最终 URL 或移除 |
| 内部 redirect chain | 内链经过一个或多个 redirect | 更新 source 直指最终 canonical |
| 重要页发现路径弱 | 与同类重要页相比路径明显差，且影响用户/抓取 | 改 hub、导航或上下文链接 |
| 模板过度链接 | 大量弱相关全站链接影响使用/抓取 | 按用户任务收缩模板 |
| 锚/目标误导 | Anchor 与目标内容不符 | 改为准确描述或换目标 |
| 重复/canonical 冲突 | 内链持续指向非首选 URL | 统一内链、redirect、canonical、sitemap |
| 发布回填欠账 | Tracker 中新页无 donor/Hub 更新 | 补齐并复验 |

不要使用“入链 <5”“深度 >3”“单页 >150”作为通用失败线。可将其作为同站异常检测的候选阈值，但必须结合模板、规模、页面类型和业务路径。

## 8. 修复优先级

1. 重要页面的孤儿、4xx、错误 canonical/redirect 目标。
2. 阻断主要用户与商业路径的问题。
3. 发布/迁移造成的大范围链接图断裂。
4. 相关 donor 对合格 acceptor 的支持缺失。
5. 模板噪声、锚文本和次要可用性优化。

优先级由影响和证据决定，不能仅按链接数量排序。

## 9. AI Search 边界

清晰的信息架构和可抓取链接仍是正常 SEO 基础。Google 没有公布“成簇互链提高 AI 引用 3.2 倍”或“内容前 30% 获得固定引用比例”的规则；删除这类无来源倍数。

为用户组织相关页面和原创证据即可。不要为了 AI 摘引把“相关阅读”机械移到底部、重复实体锚或制造 fan-out 页面。

## 10. Tracker 与复盘

`trackers/links.md` 至少记录：

```text
audit_date | crawl coverage | orphan candidates | broken/redirect edges |
priority pages | publish backfill queue | source -> target changes |
artifact verification | review metric/date | result
```

节奏：每次发布/迁移处理相关链接；当积压影响发现、重要页面出现异常、结构大改或批量发布时扩大审计。复查频率由变化速度、风险、数据成熟度和待决策事项决定，不因月度日历重复无变化的 crawl。

## 不要做

- 不发布希望参与搜索但完全无入链的页面。
- 不为固定配额添加弱相关链接。
- 不把页脚/导航视为无效，也不把正文链接神化成唯一有效类型。
- 不用全站精确锚模板操纵相关性。
- 不将内链改动直接宣称为排名或 AI 引用提升；先验证产物，再复盘一方结果。
- 不忘记记录和回滚大范围模板改动。
