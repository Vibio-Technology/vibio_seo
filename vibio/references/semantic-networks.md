# 语义内容网络：从买家任务设计页面与关系

语义内容网络是一张站内规划图：把真实产品/概念、买家任务、页面责任和内部链接放在一起，帮助团队决定应强化、合并、新建还是停止哪些页面。它不是 Google Knowledge Graph 的镜像，也不存在可通用计算的“语义覆盖率”或“主题权威分”。

## 一、输入必须有证据

优先使用：

- 产品目录、规格、测试、认证范围、案例和专家访谈；
- GSC 查询与落地页、站内搜索、销售/RFQ、客服问题和 CRM 阶段；
- 目标市场、目标语言下带日期的 SERP 与竞争页面观察；
- 现有 URL 库、内链、canonical、抓取/索引和转化路径；
- 经核验的 paid search terms，只用于了解用语、意图和落地页表现。

外部关键词、聚类、抓取或 SERP 能力是可选输入。能力缺失时，用现有页面、第一方数据、人工抽样和用户访谈完成最小可用网络；未知需求保持未知，不能由模型补造。

## 二、网络对象

### 节点

节点不是“必须有一个 URL 的关键词”，而是规划对象：

- **业务对象**：品牌、服务、产品线、型号、材料、地点；
- **属性与证据**：规格、兼容性、测试、认证、交付能力、案例；
- **任务与问题**：了解、筛选、比较、配置、排错、采购、售后；
- **受众与场景**：行业、角色、应用、地区、合规环境；
- **页面**：已存在 URL、计划页面或需要合并/退役的页面。

### 关系

使用读者能理解并能由事实支持的关系，例如：

- 产品属于某产品线，具有某项规格；
- 产品适用于某场景，但受某条件限制；
- 两种方案可比较，且决策取决于明确条件；
- 某认证或测试覆盖特定主体、流程或型号；
- 支持内容解释商业页面上的决策；
- 案例证明某种任务在特定边界内的结果。

关系名称只是内部建模语言，不表示搜索引擎采用了相同关系，也不应为了填满网络而制造内容。

## 三、给每个页面一项主责任

页面规划表至少包含：

| 字段 | 要回答的问题 |
|---|---|
| 市场/语言 | 这项需求在哪个独立市场得到验证？ |
| 买家/任务 | 谁在什么情境下要完成什么？ |
| 查询族 | 哪些真实用语表达同一任务？ |
| 主页面 | 哪个 URL 对该任务负责？ |
| 页面状态 | 保留 / 强化 / 合并 / 新建 / 重定向 / 退役 / 待验证 |
| 支持事实 | 企业能提供什么规格、经验、数据或证据？ |
| 上下游页面 | 用户此前需要什么、下一步要去哪里？ |
| 需求证据 | 第一方数据、SERP、访谈或明确标注的假设 |
| 负责人 | 谁提供事实、实施并持续维护？ |
| 结果指标 | 上线后用什么搜索或业务信号判断？ |

多个近义查询通常应由一个能完整完成任务的页面承接。只有受众、任务、页型、产品能力或市场确有差异时才拆 URL。无法提供信息增益的节点可以留在研究表中，不必发布。

## 四、内容形态由任务决定

定义、对比、规格表、选择器、案例、视频、FAQ 或排错步骤都是编辑选择，不是排名模板。选择能最清楚完成用户任务且团队能持续维护的表达方式：

| 用户任务 | 可能合适的表达 | 必须具备的事实 |
|---|---|---|
| 理解产品或概念 | 简洁说明、术语表、示意或示例 | 准确定义、适用范围、限制 |
| 比较方案 | 对比表、决策规则、实测案例 | 可比口径、条件、取舍、来源 |
| 核对适配性 | 规格/兼容表、筛选器、下载资料 | 型号、单位、测试方法、版本 |
| 评估供应能力 | 产品/服务页、工厂能力、案例 | 可兑现范围、流程、证据 |
| 解决问题 | 诊断流程、步骤、支持入口 | 症状、原因边界、安全与升级条件 |
| 采取商业动作 | 询价、配置、联系或购买路径 | 所需输入、响应预期、隐私说明 |

表格、列表、问答和视频只在提高理解或完成任务时使用。不得把内容长度、格式多样性、FAQ 数量或 Schema 当作排名或生成式 AI 展示保证。

## 五、内链拓扑

内链同时服务发现、理解和用户导航：

```text
商业主页面
  <-> 规格、证据、应用与案例
  <-> 比较、选择与排错任务
  -> 询价、购买或支持动作
```

- 每个重要页面都应从用户实际会经过的相关页面获得可抓取入口；
- 支持页在语境合适时链接到负责转化的主页面，商业页也链接到必要证据；
- 横向链接只连接确有下一步关系的页面，不为“语义关联”堆链接；
- 锚文本准确、自然地描述目标，不要求精确匹配关键词；
- 新建、合并或重定向页面时同步更新 donor、导航、sitemap 和 tracker；
- 数量、点击深度和回填节奏按站点结构与观察证据决定，不套固定密度。

具体操作以 `references/link-architecture.md` 为准。

## 六、机会排序

为每个节点或页面输出 `now / next / later / reject`，并保留决策证据。

### Now

- 修复会阻断重要页面抓取、索引、规范化或用户路径的网络问题；
- 解决多个页面争夺同一任务、主页面不清或重要页面成为孤儿的问题；
- 强化已有需求和商业价值均得到证明、且企业有独有信息的页面。

### Next

- 补齐完成已验证购买任务所需的规格、比较、应用、案例或支持内容；
- 合并重复节点，重映射查询族和内部链接；
- 在已上线页面获得信号后，根据真实查询和销售问题扩展。

### Later

- 需要更多需求证据、专家材料、开发、翻译、法务或测量基础的机会；
- 只有在前置页面和数据证明价值后才合理投入的高成本资产。

### Reject

- 需求来自直译或工具建议，但目标市场 SERP/买家任务不支持；
- 企业无法兑现承诺或提供独有信息；
- 与现有页面任务相同，拆分只会制造重复；
- 主要目的只是填充实体、关键词或内容格式配额。

第三方 KD、搜索量或竞争分可以作为注明供应商、市场和日期的辅助证据，不能成为普遍门槛。排序方法与 `references/authority-cascade.md` 保持一致。

## 七、验证与学习

### 产物验证

- 页面责任、canonical、重定向和 sitemap 没有互相冲突；
- 渲染后的链接可抓取，指向预期规范 URL；
- 页面上的关系有事实支持，且不同页面没有关键口径冲突；
- 每个发布任务含 donor 回填、负责人、完成定义和 fallback。

### 结果观察

- 查询是否落到预期页面，是否仍存在轮换或错配；
- 相关曝光、点击和目标市场的可见性如何变化；
- 用户是否继续到规格、案例、询价、购买或支持步骤；
- 自然落地页带来的合格线索或销售结果是否变化；
- 内容维护、重复和抓取成本是否降低。

按数据成熟度设复核窗口，并与未改页面、历史可比区间或页面 cohort 对照。构建完成、覆盖图更大或页面数量增加只证明产物发生变化，不证明 SEO 或 AI 可见性改善。

## 八、模式连接

| 模式 | 使用方式 |
|---|---|
| KEYWORD | 将经独立市场验证的查询族分配给主页面和买家任务 |
| WRITE | 在 brief 中写明页面责任、证据缺口、上下游页面和完成定义 |
| AUDIT | 找主页面冲突、孤儿、重复、事实矛盾和任务缺口 |
| FIX | 实施 URL、canonical、重定向、导航和内链修复并验证产物 |
| REVIEW | 按页面与市场复盘查询归属、业务结果和继续/合并/停止决策 |
