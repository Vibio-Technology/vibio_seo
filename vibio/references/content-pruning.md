# 内容库存治理：保留、改进、合并与下线

内容修剪是 URL 级的信息架构和资产治理，不是通过删除固定比例页面来获得“整站质量加分”的技巧。删除页面可能改善用户路径和维护效率，也可能损失需求覆盖、引荐、链接、合规信息和收入；每项动作都需要明确机制与证据。

## 一、原则

- 先确认页面的用户任务、业务作用和目标市场，再看流量。
- 零点击、页面年龄、字数、外链数或工具评分都不能单独决定删除。
- 不声称 Google 核心更新会因为“剪掉弱页”普遍奖励整个网站。
- 对同一页型和市场使用可比窗口，并考虑季节性、销售周期与 GSC 数据限制。
- 先处理错误、垃圾内容政策风险、泄露或合规问题；此类动作不必等待 SEO 实验。
- 高风险批量操作必须分批、可验证、可回滚，不能按固定页面比例推进。

## 二、建立库存真相

每个可索引或可发现 URL 至少记录：

```text
URL / canonical / status:
Page type / locale / target market:
Primary user task and query class:
Business purpose / owner / required retention:
Index and crawl evidence:
GSC page/query evidence and date range:
Organic landing and qualified business outcome:
Internal links / external referrals / verified backlinks:
Accuracy, originality and last material change:
Overlap or cannibalization candidates:
Dependencies: hreflang, feeds, PDFs/media, campaigns, support/legal:
```

数据窗口应足以覆盖该页面的自然需求与业务周期；没有 GSC、分析或 CRM 数据时明确标记缺口。先按页型、主题、市场和模板分组，再比较组内表现，避免用全站统一分数淘汰小众但必要的页面。

## 三、判定页面的真实角色

逐项回答：

1. 页面是否完成一个仍存在的用户或业务任务？
2. 内容是否准确、原创、可维护，并符合当前目标市场？
3. Google 是否选择该页服务相关查询，还是选择了另一个 URL？
4. 它是否产生合格访问、转化、支持价值、引荐或已核验链接？
5. 它与其他页面是互补关系，还是在争夺同一主要任务？
6. 下线后用户、搜索引擎、合作伙伴、销售和合规流程会去哪里？

不能回答时先取证，不用平均分、固定流量档位或“年龄风险分”代替判断。

## 四、决策矩阵

| 决策 | 证据条件 | 实施要点 |
|---|---|---|
| 保留 | 任务独立、内容有效，且仍有用户/业务/合规价值 | 记录维护负责人和触发复核的信号 |
| 改进 | 任务仍成立，但事实、意图、证据、可用性或转化路径存在明确缺口 | 以诊断机制为范围，保存前后规格并验证 |
| 合并 | 多页服务同一主要任务，存在重叠且一个主页面能完整承接 | 选择主页面、迁移独特价值、永久重定向并更新所有引用 |
| 保留但不索引 | 页面对用户/流程必要，但没有独立搜索用途且移出索引符合策略 | 谨慎使用 `noindex`，确保不依赖 canonical 传递信号，并验证导航影响 |
| 下线 | 任务消失、内容无保留义务且没有真正等价替代 | 返回 404 或 410，移除 sitemap 和内部入口 |
| 永久重定向 | URL 已移动或存在满足同一主要任务的等价/合并页 | 使用服务端 301/308，直接指向最终相关目标 |
| 暂不决策 | 数据不成熟、季节未覆盖或依赖不清 | 标记缺失证据和下一决策条件，不把等待写成保留结论 |

不要把无等价替代的 URL 批量重定向到首页、分类页或“最接近”的商业页；这会误导用户并可能被视为 soft 404。有链接也不自动要求重定向，先人工核验来源、流量、相关性和目标页承接能力。

## 五、合并协议

1. 以用户任务、目标市场、内容完整性、历史查询、业务价值、内部发现和已核验引荐共同选择存活页，而不是只看外链数或平均排名。
2. 建立 section-level 映射，把被合并页中仍准确且独特的证据、案例、媒体和答案迁入主页面。
3. 确认合并后不会混合不同意图、语言、地区、产品状态或法律要求。
4. 部署服务端永久重定向，避免新产生的链和循环；历史链应尽可能改为直达最终目标。
5. 更新站内链接、canonical、hreflang、结构化数据、导航、feed、sitemap 以及外部可控引用。
6. 保存旧 URL、目标 URL、原因、部署日期和回滚映射，并抓取验证状态码与渲染内容。

发布后通过准确 sitemap 和可抓取内链帮助发现变更。不得把普通页面批量重新提交或调用 Indexing API 当作合并完成条件；该 API 仅适用于 Google 当前支持的职位和直播场景。

## 六、分批执行与基线

高风险库存治理应先选具有代表性的处理 cohort，并尽可能保留匹配对照。批次大小取决于站点规模、模板共享范围、收入风险、回滚能力和测量 power，不预设首次删除比例。

实施前记录：

- 处理页与目标页合计的查询、展示、点击和目标市场分布；
- 自然落地页的合格转化、销售管道或其他业务价值；
- 索引/canonical 状态、内部入口、引荐和已核验链接；
- 同类未处理页面的基线和历史波动；
- 迁移、改版、促销、季节和算法变化等混杂因素；
- 每项动作的预期机制、护栏与回滚条件。

不要只记录“全站索引页数”或抓取总量。更少的索引页和更多抓取都不是业务成功本身，必须连接到目标 URL 的发现、搜索需求和合格结果。

## 七、上线 QA 与复盘

### 实施层

- 保留页内容、标题、canonical、hreflang 和结构化数据符合规格；
- 下线页返回计划状态，重定向直达相关最终页，无链、环或批量 soft 404；
- sitemap、导航、内链、媒体、feed 和分析标记已同步；
- 关键用户旅程、目标市场和移动/桌面渲染可用。

### 结果层

按搜索系统实际抓取和数据成熟度设置复核点，而不是固定 30/60/90 天：

1. 先确认部署与测量有效；
2. 再确认旧 URL、目标页和 Google-selected canonical 的变化；
3. 对处理页与目标页的合并 cohort 比较查询覆盖、相关曝光和点击，不能只看目标页单独增长；
4. 与对照/历史趋势比较自然落地页的合格业务结果和护栏；
5. 若无可比基线或干扰过多，结论标为 `inconclusive`，不宣称修剪提升了整站质量。

采用 `references/review-engine.md` 的 verdict。出现误重定向、重要查询/业务价值损失、合规信息缺失或用户路径破坏时，按预定条件回滚或修正。

## 八、可选能力与降级

| 所需证据 | 可选能力 | fallback |
|---|---|---|
| 全站库存/渲染 | `seo-firecrawl`、`seo-drift` | 从 sitemap、CMS 导出和服务器路由建清单；对风险分层抽样 |
| GSC/索引 | `seo-google` | 请求 GSC 导出；不可得时只报告技术产物和公开发现信号 |
| 外链 | `seo-backlinks` | 使用分析引荐、用户导出和人工核验来源 URL |
| SERP/重叠 | `seo-dataforseo` | 结合 GSC page-query 数据与目标市场有限人工抽样 |
| 业务价值 | GA4/CRM | 请求落地页 cohort 导出；不可得时不作收入结论 |

所有外部能力均为可选。缺失时缩小结论范围，绝不虚构页面表现、外链价值、索引或转化。

```markdown

| Expected mechanism | Dependencies | Deploy date | Artifact QA | Search result |
| Business result | Verdict | Rollback/next action | Owner |
```
