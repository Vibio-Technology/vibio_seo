# 来源、第一手经验与可信内容

本文件支撑 WRITE：先把论断与证据逐条绑定，再写出能完成用户任务、可由专家核验的页面。来源数量、E-E-A-T 标签、内容格式、Schema 和所谓“AI 可摘引结构”都不是排名或 AI 引用的代理指标。

遵守 `references/evidence-policy.md` 与 `references/google-search-docs.md`。Google 将 E-E-A-T 用作内容自评框架，Trust 最重要；它不是可单独测量的分数，也不能据此承诺排名。

## 一、证据表先于正文

从 brief/大纲提取所有需要外部或第一方支持的论断，包括：

- 数字、比例、价格、容量、性能和市场规模；
- 标准、法律、合规、安全、健康和兼容性要求；
- “优于”“适合”“导致”“降低”“最快”等比较或因果结论；
- 客户结果、内部测试、交付能力、认证和制造范围；
- 会影响购买、投资或风险判断的专业建议。

证据表使用以下字段：

```md
| Claim ID | 计划中的准确论断 | 论断类型 | 原始来源 / URL | 原文或数据位置 | 发布/版本日期 | 适用范围与方法 | 限制/利益关系 | 核验日期 | 正文处理 |
|---|---|---|---|---|---|---|---|---|---|
| C-01 | | | | | | | | | 使用 / 限定 / 待专家确认 / 删除 |
```

URL 可打开不等于论断已核实。核验必须确认：页面确实包含该信息、上下文没有改变含义、版本/样本/市场适用、来源主体有能力支持该类主张，并记录精确位置。

无法核验时：

- 不写；或
- 改成范围准确、来源允许支持的较弱表述；或
- 明确标为估算/观点，并交代依据、方法和限制；或
- 在发布状态中标为“等待专家/法务/数据确认”。

绝不能生成看似可信但未经打开核对的 URL、数字、引文、标准条款或客户结果。

## 二、按论断选择来源

来源是否合适取决于论断，不靠一个万能等级表：

| 论断 | 优先来源 | 核验重点 |
|---|---|---|
| 搜索功能、资格或平台行为 | 当前官方文档 | 最新状态、适用产品、核验日期 |
| 标准与法规 | 发布机构、监管方、正式标准文本 | 版本、司法辖区、强制/建议边界 |
| 科学或技术因果 | 原始研究、系统综述、标准测试方法 | 样本、方法、材料/条件是否可比 |
| 市场规模与趋势 | 有方法说明的原始数据或正式报告 | 地区、年份、定义、样本、预测假设 |
| 产品性能 | 当前规格书、测试报告、可复核内部数据 | 型号、单位、方法、批次、适用条件 |
| 客户或项目结果 | 经授权的一手记录 | 基线、范围、时间、归因限制和隐私 |
| 实施建议 | 官方要求、项目实验或专家明确署名经验 | 是规则、观察还是待验证假设 |

二手文章可用于发现原始出处或补充背景，但不能因为域名知名就自动成为“权威证据”。制造商或企业自己的资料适合证明其产品和流程，不应被用来无条件证明整个行业优越性。来源年份是否过旧取决于论断变化速度；不要套固定“近几年”规则。

### 来源冲突

- 先检查定义、版本、地区、样本和测量方法是否不同；
- 政策/资格采用最新官方来源；
- 不能消解的冲突要在正文说明，或删除确定性结论；
- 不挑选最有利数字而隐藏同等质量的反证；
- 页面更新时重新核验易变论断，不只检查链接是否存活。

## 三、第一手经验的使用边界

第一手经验能提供竞争页面没有的信息，但必须可验证且不越权：

- 写明谁观察、在什么条件下、用什么方法、针对什么型号/市场；
- 内部测试保留单位、样本、设备/方法、日期和限制；
- 案例写清基线、实施内容、结果口径、同期因素和授权范围；
- 专家意见点明专家角色和判断依据，不用“业内认为”代替来源；
- 失败、例外和不适用条件与正面结果同样重要；
- 客户、员工和业务数据先经过授权、隐私和商业保密审查。

内部数据不能因“第一方”身份自动获得高置信度。方法不透明、样本不适用或存在明显利益关系时，要降低结论强度并披露限制。

## 四、把 E-E-A-T 落到页面事实

### Experience

在相关处展示真实使用、制造、测试、交付或解决问题的经历，包括条件、结果和限制。没有第一手材料时不要虚构“我们发现”。

### Expertise

使用准确术语、单位、决策逻辑和安全边界。需要专业资质的主题由具备相应能力的人审核，并如实说明审核角色。

### Authoritativeness

让作者/组织为何能讨论该主题变得可核验：真实身份、职责、作品、资质和一手工作。外部提及、链接或资料页可以作为背景证据，但其数量、域名分数或是否进入某数据库不等于排名信号。

### Trust

- 事实、来源、发布日期/更新日期和责任主体清楚；
- 商业利益、测量限制、适用条件和不确定性透明；
- 公司、联系、退换/隐私、价格或询价流程与业务一致；
- 页面不冒用客户、认证、标准、评价或专家身份；
- 更正机制明确，过期内容会被更新、合并或撤下。

E-E-A-T 检查用于发现可信度缺口，不把四项相加成分数，也不声称“增加作者框”本身会提升排名。

## 五、写作与结构

### 先完成用户任务

- 开头尽快回答页面的主任务，但答案长度和位置按任务复杂度决定；
- 定义关键术语，明确产品、标准、地区和条件，避免代词造成歧义；
- 比较内容使用同一口径，给出取舍和不适用场景；
- 规格、步骤和对比在表格/列表更清楚时就使用；复杂判断需要叙述时不要强行列表化；
- 每一节都应增加事实、判断或行动价值，删掉不改变理解的填充段落；
- 内链连接真实的前后任务和证据页，数量由信息架构决定。

内容形态服务可读性和决策，不是搜索引擎偏好的固定模板。无需让每个标题都变成问题、每段都能脱离上下文，或为获得 AI 引用而拆成极小片段。

### 关于生成式 AI 搜索

Google 当前说明 AI Overviews/AI Mode 建立在核心 Search 系统和正常技术资格之上，不需要专用 AI 改写、特殊 Schema、`llms.txt` 或人为制造提及。清楚、原创、可核验、对用户有帮助的页面仍是重点，但任何写法都不能保证被 AI 系统引用。

对其他平台，只能依据该平台当前官方说明或带日期的可重复观察提出实验。表格、Q&A、摘要或“answer-first”可以因用户体验采用，不能宣称它们天然更易被引用。`FAQPage` 等标记只在当前受支持功能且页面真实符合要求时使用；Schema 提供机器可读线索并可创造功能资格，但不保证排名或生成式 AI 展示。

## 六、去除低质量 AI 文风

无论是否使用生成工具，都做以下编辑：

- 将“研究表明”“专家认为”“被广泛采用”替换为具体来源和边界；
- 删除“行业领先”“革命性”“至关重要”等无事实支持的宣传词；
- 删除预告、总结性套话、伪金句、同义改写循环和凑数列举；
- 把抽象好处改成机制、条件、规格、案例或可执行判断；
- 统一同一实体、单位和术语，不为语言变化牺牲准确性；
- 检查模型是否补造客户经历、数据、功能、链接或不存在的出处；
- 保留真正承载规格、步骤、比较和风险的信息结构，不把技术内容洗成空泛散文。

“像人写”不是质量标准；可核验、准确、具体、适合目标读者才是。

## 七、发布审查

审查不使用固定 Agent 数、固定轮次或固定分数。根据页面风险安排编辑、主题专家、产品、法务或本地语言审核，直到阻断问题关闭或明确标为不发布。

### 阻断项

- 核心意图或页面承诺与企业能力不符；
- 重要事实、数字、引文、标准或客户结果无法追溯；
- 来源不支持正文强度，或忽略关键限制/反证；
- 医疗、法律、安全、财务、合规或性能主张缺适格审核；
- 内容泄露个人信息、客户机密或未经授权的品牌/评价；
- 页面主要是批量改写，缺少对目标用户的实质价值；
- 语言、单位或市场规则错误，可能误导用户。

### 质量检查

- 页面确实完成目标市场买家任务；
- 相比现有页面/竞争结果，信息增益具体且有证据，不要求固定条数；
- 每个重要论断都能回到证据表，版本和适用范围正确；
- 第一手材料已真实落地，并披露方法与限制；
- 标题、摘要、正文、CTA 和内部链接表达同一页面责任；
- 表格、列表、视频、Q&A 等结构都有用户价值，不用于凑格式；
- 作者/审核者、更新日期和商业关系只在真实且有帮助时展示；
- 需要持续更新的论断有 owner 和复核触发条件。

审查结论使用：`ready / needs-evidence / needs-expert-review / needs-legal-or-privacy-review / do-not-publish`，并列出负责人和完成定义。

## 八、能力路由与 fallback

网络检索、网页读取、论文/标准数据库、SERP 和事实核查能力都是可选。调用前检查可用性：

- 能力存在：打开原始来源并记录准确论断、范围和核验日期；
- 能力缺失：优先使用用户提供的原始文件、项目资料和已核验缓存；
- 只有摘要或二手转述：标为未完全核验，不升级为确定性事实；
- 无法核验且会影响决策：阻断该论断或向用户索取来源；
- 任何情况下都不得编造 URL、工具结果、访问状态或引用内容。

## 九、最小交付物

WRITE 交付至少包含：

- 页面任务、目标市场/语言和主受众；
- 证据表及未解决的来源缺口；
- 正文/页面组件与事实对应关系；
- 需要专家、产品、法务、隐私或母语审核的项目；
- 内链与 CTA 计划；
- 发布前状态、负责人、完成定义和 fallback；
- 上线后的产物验证与结果指标。
