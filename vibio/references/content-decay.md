# 内容衰退诊断与恢复

内容衰退不是“页面变旧”的同义词。只有在可比口径下出现搜索或业务结果变化，或页面事实、任务与目标市场已经失效时，才把页面列为衰退候选。发布日期、固定跌幅、平均排名或竞争对手更新日期都不能单独证明衰退。

## 一、先定义页面任务与可比基线

为每个候选 URL 记录：

- 页面任务、目标词族、目标国家/语言、页型和主转化；
- canonical、索引状态、模板、主要内部入口和最近一次实际部署；
- GSC 的页面、查询、国家、设备、搜索类型和日期口径；
- 自然落地页的参与、转化、合格线索或收入口径；
- 发布与改动历史、内容来源、产品/法规/标准的有效期；
- 同期迁移、模板改版、促销、品牌活动、算法变化和需求季节性。

比较窗口由该站自身的数据量、搜索季节和业务周期决定。优先使用同比、匹配页面组或稳定的改动前趋势；没有可比基线时标记 `no-comparable-baseline`，只做事实和实施审查，不宣称页面发生了 SEO 衰退。

## 二、发现候选，而不是用阈值定罪

1. 按页面组、市场和查询类别建立项目自身的正常波动范围。
2. 标记明显偏离自身历史或可比页面组的 URL，同时保留原始分母。
3. 将品牌词/非品牌词、国家、设备和查询组合拆开，避免总量掩盖意图变化。
4. 检查内容是否存在已失效的规格、价格、法规、接口、产品状态或来源。
5. 对重要候选保存带日期和地区的 SERP、页面与渲染快照，再进入诊断。

检测频率应由页面变化速度、抓取节奏、数据量和商业风险决定。快变信息可以更频繁复核，稳定内容只需在新证据出现时复核；不得用统一的周、月、季度或页面年龄规则覆盖所有项目。

## 三、先诊断机制

| 观察 | 需要排查的机制 | 最小验证证据 |
|---|---|---|
| 多个可比页面和市场同时下降 | 季节、需求、品牌或全站技术变化 | 同比/对照 cohort、趋势数据、部署记录 |
| 单页展示下降 | 抓取/索引、canonical、查询需求、意图或竞争变化 | URL Inspection、查询分布、目标市场 SERP |
| 点击变化但展示相近 | 查询/设备组合、实际标题摘要、SERP 形态或品牌需求变化 | 分层 GSC 与带日期 SERP 抽样 |
| 平均排名变化 | 查询组合变化、页面互相替代或真实位置分布变化 | 查询级分布与 page-query 映射；不能只看平均值 |
| 同主题页面一升一降 | cannibalization、Google 选择了不同主页面，或正常替代 | 同一查询集的页面分布、canonical 与内链 |
| 搜索访问稳定但业务结果恶化 | 流量意图、报价、库存、表单、销售跟进或测量变化 | 自然落地页 cohort、CRM 阶段和埋点 QA |
| 页面事实过期或可能误导 | 内容正确性和合规风险 | 最新一方资料、官方标准或产品专家确认 |

SERP 新增功能与点击下降可以相关，但不能仅凭两者同时出现就认定功能“抢走了点击”。竞争对手内容更长或日期更新，也不能证明其胜出原因。

## 四、选择动作

| 决策 | 适用条件 | 完成定义 |
|---|---|---|
| 保持/观察 | 页面仍准确，变化可由需求或正常波动解释 | 记录不改原因、下一观察信号和窗口 |
| 纠错 | 事实、法规、价格、规格或可用性已经错误 | 专家/官方来源核验，错误内容和相关链接已修复 |
| 定向更新 | 已确认的信息缺口、意图错配或展示表达问题 | 只改与机制有关的内容，并保存前后规格 |
| 重构 | 页面任务或目标市场已经变化，局部修补不足 | 新任务、结构、证据、内链和转化路径均通过 QA |
| 合并 | 多个 URL 服务同一主要任务且分散信号或维护成本 | 有价值内容迁移，相关旧 URL 永久重定向，内链更新 |
| 下线 | 页面已无用户/业务/合规价值且无等价替代 | 返回 404/410；只有存在真正相关替代页时才永久重定向 |
| 受控实验 | 机制仍不确定，但改动可逆且结果会改变决策 | 有处理/对照、主指标、护栏和停止条件 |

优先级综合业务暴露、伤害大小、证据置信度、投入、依赖和可逆性。不要使用 `流量 × 跌幅 × 商业价值` 之类没有校准的分数，也不要机械优先某个排名区间。

## 五、更新实施规范

- 先修正能影响用户决策的错误，再补充目标市场真正缺失的一手证据、比较、规格或案例。
- 保留仍然准确且有用的历史来源；“最新”不等于“更可靠”。注明适用日期、版本和市场。
- 让 title、主标题、正文和转化承诺与当前任务一致，不采用固定字符数或关键词密度。
- 只有当前支持且与可见内容一致时才更新结构化数据；它只提供相应搜索功能资格。
- 更新相关内部入口、hreflang、canonical、媒体和外链，并验证渲染后产物。
- 只有发生对用户有意义的实质变化时才展示真实的“最后更新”日期。不得只改日期、年份或少量措辞来制造新鲜感。
- sitemap 的 `lastmod` 仅反映重要内容变更。发布后依靠可抓取内链和准确 sitemap 发现页面，不批量或反复提交普通 URL。
- Google Indexing API 只用于当前官方支持的 `JobPosting` 和直播 `BroadcastEvent` 场景，不是普通内容刷新工具。

每次更新写入 `.vibio/changelog.md`：改动范围、预期机制、目标市场、部署日期、主指标、护栏、观察条件和回滚方案。

## 六、验证增量

按 `references/review-engine.md` 和 `references/seo-experimentation.md` 复盘：

1. **实施**：真实响应和渲染结果与规格一致。
2. **抓取/索引**：搜索系统已经抓取目标版本，canonical 与索引状态符合预期。
3. **搜索**：在相同页面、查询、国家、设备和品牌口径下比较处理组与对照/历史趋势。
4. **业务**：观察自然落地页 cohort 的合格转化、销售管道或收入护栏。

观察窗口从实际部署和搜索系统有机会看到变更后开始，由项目历史抓取延迟、数据成熟度和预先定义的最小有意义效果决定。没有对照或稳定趋势时，只能报告方向；不得把时间上的先后关系包装成恢复成功。

结论使用：`implementation-failed`、`not-yet-observable`、`directional-positive`、`incremental-positive`、`no-detectable-change`、`negative-or-regression` 或 `inconclusive`。

## 七、可选能力与降级

| 所需证据 | 可选能力 | 不可用时的 fallback |
|---|---|---|
| GSC/索引 | `seo-google` | 请求用户导出；仍不可得时只做公开可抓取信号与实施审查 |
| 抓取/渲染 | `seo-firecrawl`、`seo-drift` | 从 sitemap 取范围明确的 URL，用浏览器/curl 抽样并保存快照 |
| 实时 SERP | `seo-dataforseo` | 在目标地区做带日期的有限人工抽样，明确不完整 |
| 外链/引荐 | `seo-backlinks` | 使用已提供的导出、分析引荐和人工核验来源页 |
| 业务结果 | GA4/CRM | 定义所需导出与埋点；不可得时不评价收入增量 |

外部能力缺失不会阻止范围明确的诊断，但绝不能据此编造排名、查询、索引、流量、外链或转化数据。
