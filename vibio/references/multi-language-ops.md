# 多语言 SEO 运营：每个市场独立验证

语言版本不是源站内容的复制任务。每个“国家/地区 × 语言 × 买家 × 主要搜索渠道”都是独立市场单元，需要自己的需求、SERP、竞争、合规、内容证据和业务测量。

## 一、市场立项

对每个候选市场记录：

| 决策维度 | 所需证据 |
|---|---|
| 商业可服务性 | 产品可售范围、物流/服务、币种、法规、销售支持 |
| 买家与转化 | 目标角色、采购任务、主转化、销售周期和不服务范围 |
| 本地需求 | 本地 RFQ/CRM、站内搜索、GSC、paid search terms、访谈或本地关键词数据 |
| SERP 与渠道 | 目标地区、目标语言下的结果页、主流搜索渠道和页型 |
| 内容能力 | 母语研究/编辑、产品专家、法务、客服和持续维护能力 |
| 测量能力 | 可按市场/语言观察曝光、自然落地页和合格业务结果 |

如果主要证据只是“其他语言市场表现不错”或“应该有某语言版本”，状态应为待验证。不要用全球搜索量、另一国家的 SERP 或源语言转化率代替目标市场证据。

## 二、关键词与意图必须独立研究

翻译可以生成研究种子，不能完成关键词验证。每个市场都要重新执行：

1. 从该市场的销售用语、询盘、客服、论坛/行业资料、站内搜索和真实 search terms 收集母语表达；
2. 在目标位置和语言下采样 SERP，判断搜索者、意图、结果类型和竞争对象；
3. 将词族映射到该市场已有页面、新页面、合并页面或拒绝项；
4. 核对企业在该市场能否兑现页面承诺，并能否提供本地适用的证据；
5. 按 `references/keyword-validation.md` 保留来源、日期和不确定性。

同一产品在不同市场可能需要不同页面任务、示例、单位、法规说明和转化路径。不得把源市场的内容清单、优先级、KD 分档或发布顺序直接复制过来。

## 三、URL 与交付架构

优先使用稳定、可抓取、可分享的独立 locale URL。子目录、子域和 ccTLD 都可以工作，选择取决于所有权、基础设施、地区定位、法规、团队权限和维护成本，而不是普遍的“权重更高”结论。

| 方案 | 适合考虑的情形 | 主要风险 |
|---|---|---|
| 子目录 | 同一平台、同一域名治理，团队可统一部署 | 模板和权限必须支持市场差异 |
| 子域 | 技术栈、团队或发布系统独立 | 监测、治理和跨系统一致性成本 |
| ccTLD | 业务明确按国家独立，能承担本地运营 | 域名、内容、合规和维护投入更大 |

不要只靠 IP、Cookie 或浏览器语言在同一 URL 动态替换主要内容；搜索爬虫和用户可能看不到所有版本。自动跳转应允许用户切换并避免阻断爬虫。语言选择器应使用普通可抓取链接，目标标签用目标语言自称更清楚。

## 四、`hreflang` 与 canonical 的事实边界

两者职责不同：

- canonical 用于表达重复或近重复 URL 中偏好的规范版本；
- `hreflang` 用于说明可索引页面之间的语言/地区替代关系；
- 它们是信号，不保证抓取、索引、排名或指定版本一定展示；
- `hreflang` 不会修复内容质量、错误语言、索引阻断或错误 canonical。

### 实施规则

- 只把内容等价、面向同一任务的 locale 页面放进同一替代集合；
- 每个集合列出自身和其他有效替代页，并保持回链；
- 使用完整绝对 URL；按 Google 当前 hreflang 支持规则验证代码：语言使用受支持的 ISO 639-1 两字母代码，地区使用 ISO 3166-1 Alpha-2，适用时脚本使用 ISO 15924。不能因为某个任意 BCP 47 变体或三字母语言码语法有效，就假设 Google 会接受；地区只在确实面向该地区时添加；
- `x-default` 是可选的无特定语言/地区 fallback，不是必须项；
- HTML、HTTP header 或 sitemap 任选适合架构的一种实现即可，避免多处维护产生冲突；
- 替代 URL 应返回可索引内容，并与 robots、状态码和 canonical 保持一致；
- 真正独立的翻译/本地化页通常各自 self-canonical，不要把所有语言 canonical 到源语言页；
- 若同一语言的地区页高度近似并计划跨 URL canonical，先确认哪些页面仍需独立进入搜索，再设计 `hreflang` 集合，避免同时发送“合并”和“独立展示”的冲突信号。

Google 的 canonical 选择与 `hreflang` 处理最终由系统决定；页面声明是输入信号，不是控制命令。Search Console 报告的 Google-selected canonical 与声明不一致时，应先查内容重复、重定向、站内链接、sitemap 和其他规范化信号，而不是仅重复添加标签。

上线前按 Google 当前的本地化版本官方文档重新核验代码与实现方式：

- https://developers.google.com/search/docs/specialty/international/localized-versions
- https://developers.google.com/search/docs/crawling-indexing/consolidate-duplicate-urls

外部 hreflang 检查能力存在时可用于批量发现缺失回链、错误代码和非成功 URL；能力缺失时抽取页面或 sitemap 中的实际标记，逐组核对。绝不能编造验证结果。

## 五、本地化生产

以目标市场 brief 为起点，而不是以源文逐句替换为起点：

- 保留必须一致的产品事实、型号和证据边界；
- 重写本地买家任务、术语、标题、导航、示例、单位、币种和 CTA；
- 核验当地标准、法规、交付、保修、隐私和法律表述；
- 只使用目标市场可兑现的案例、资质和联系方式；
- 由具备目标语言与行业能力的人审核准确性、自然度和承诺范围；
- 页面质量与信息增益不足时，宁可暂不发布，也不为覆盖语言生成批量近重复页。

机器翻译本身不能被简单判为“必然降权”。风险来自错误、低价值、无人审核或主要为操纵搜索而大规模生产的内容。自动化可以协助初稿或术语一致性，但发布标准由页面用途、风险和人工验证决定。

## 六、市场级 QA

### 技术

- locale URL 返回预期状态并渲染正确语言；
- canonical、`hreflang`、robots、sitemap 和重定向没有冲突；
- 语言选择器、导航、面包屑和内链不会把用户送回错误市场；
- 标题、摘要、开放图谱标签和适用结构化数据与可见页面语言一致；
- 表单、感谢页、事件与 CRM 能保留市场/语言归属。

### 内容与商业

- 关键词和意图由该市场独立验证；
- 价格、单位、法规、交付、联系方式和资质范围准确；
- 关键术语自然，不残留源语言占位或错误自动翻译；
- 页面回答本地买家任务，不只是源页面的语言镜像；
- 当地负责人认可承诺并能持续维护。

## 七、分市场测量

按市场/语言 URL 集合分别建立基线：

- 抓取、索引、canonical 选择和 locale 展示错误；
- 查询、曝光、点击、落地页和品牌/非品牌趋势；
- 自然落地页后的参与、表单、有效线索、销售接收和成交；
- 错误语言访问、语言切换、跨市场跳失和支持问题；
- 页面 cohort 的发布、刷新、合并与停止结果。

GSC 与分析/CRM 只能按共有聚合维度结合，不能确定性连接查询与个人转化。复核窗口由搜索量、抓取速度和销售周期决定，不固定按月或季度。

## 八、优先级与 fallback

使用 `now / next / later / reject`：

- `now`：阻断索引或把用户送错市场的问题，以及已有业务和需求证据的核心页面；
- `next`：完成已验证买家任务所需的本地证据、支持页和测量；
- `later`：依赖本地研究、翻译能力、法规确认或基础设施的扩展；
- `reject`：只有直译需求、无法服务、无法维护或会制造重复的市场/页面。

关键词、SERP、翻译、抓取和 hreflang 外部能力均为可选。缺失时使用第一方数据、用户提供资料和人工抽样；没有母语审核或市场证据时，把发布任务降级为研究/待审，不假装已本地化完成。

## 九、常见错误

- 直接翻译关键词、页面地图和优先级；
- 用一个国家或全球数据代表所有同语言市场；
- 将所有 locale 页面 canonical 到源语言，又要求它们独立展示；
- 把 `hreflang` 当成去重、收录或排名保证；
- 自动跳转导致爬虫或用户无法访问其他版本；
- 复制资质、价格、交付和案例，却没有确认本地适用范围；
- 用固定内容格式、字数或 Schema 代替本地买家价值；
- 混合市场数据后宣称国际化有效。
