# 实战学习回路：从项目观察形成待审假设

`.vibio/` 保存当前项目状态；跨项目学习库只在用户明确同意后启用。它保存的是经过最小化和去标识化的项目观察，不是自动改写方法论的“经验真理”。

学习回路必须遵守 `references/evidence-policy.md`：相关性不证明因果，单次成功/失败不能推广，多个观察也只能形成有适用边界的待审假设。任何 reference 修订都需要人工审查和明确批准。

## 一、启用边界

默认行为：

- 只读写当前项目的 `.vibio/`；
- 不创建项目外学习库；
- 不把一个客户的数据用于另一个客户；
- 不因已有全局文件存在就自动读取。

只有用户明确授权跨项目积累时，才确认：存储位置、允许保留的字段、可复用范围、访问者、保留/删除方式，以及是否允许读取历史观察。授权不明确时继续使用项目内状态，不阻断当前 SEO 工作。

## 二、什么值得记录

只记录同时满足以下条件的观察：

- 决策、预期机制、实施范围和日期可追溯；
- 结果来自目标项目的第一方数据或范围明确的项目实验；
- 观察窗口与业务/搜索周期相符，且定义前后一致；
- 已记录主要混杂因素、同期改动和数据缺口；
- 结论可能改变下一次同类决策；
- 内容可在不保留客户身份的情况下复用。

构建成功、按预期完成任务、单次排名波动、单次 AI 提及或只有第三方分数变化，通常不构成可复用学习。AI/搜索平台的可见性观察必须记录平台、市场、账号状态、提示/查询、日期和采样限制，且不能自动归因于某项改动。

## 三、观察条目

建议使用 append-only 文件或同等可审计存储。字段如下：

```md
## OBS-<匿名编号> — <范围明确的一句话观察>
- 记录日期：
- 匿名项目类别：<行业 / 站点类型 / 市场，不含公司或域名>
- 适用边界：<国家、语言、页型、业务阶段、流量条件等>
- 决策与改动：
- 预期机制：
- 证据等级：E2 / E3
- 数据源与定义：<保留聚合口径，不保留用户级明细>
- 对照或比较方法：
- 观察窗口：<为何足以判断>
- 实际结果：
- 同期改动 / 混杂因素：
- 可替代解释：
- 下一步验证：
- 隐私检查：通过 / 待处理
```

没有 E2/E3 项目证据的观点可以留在当前项目的假设或研究队列，但不进入跨项目观察库。不要把 E4 行业文章或 E5 专家判断伪装成“项目经验”。

## 四、从观察到候选假设

只有出现**多次、有证据、方向相近且具有可比边界的项目观察**时，才能建立候选假设。不得设置一个脱离数据质量的固定次数门槛；审查者要判断观察是否真正独立、结果定义是否一致、窗口是否成熟，以及同一来源是否被重复计算。

候选假设格式：

```md
## HYP-<编号> — <可验证、有限定条件的假设>
- 状态：pending-review
- 适用边界：
- 支持观察：<多个 OBS 编号>
- 反向 / 无效观察：
- 证据质量：<独立性、可比性、测量限制>
- 可能机制：
- 可替代解释：
- 建议验证：<如何在新项目中检验，而不是直接套用>
- 可能影响的 reference：<文件与章节>
- 隐私复核：
```

以下情况不能升级：

- 只有一次项目观察，或多条记录实际来自同一事件/同一数据；
- 结果只是相关变化，没有合理对照且同期改动众多；
- 市场、页型、意图或指标定义不同，却被强行合并；
- 只观察到排名、第三方分数或 AI 引用，没有业务/搜索证据和采样边界；
- 需要暴露客户身份才能解释结论；
- 存在同等质量的反向证据但尚未分层分析。

## 五、人工审查与方法论变更

候选假设默认状态永远是 `pending-review`，不能自动覆盖 reference，也不能自动成为新项目的决策规则。

审查时回答：

1. 支持观察是否独立、可核验且适用边界相近；
2. 是否存在未记录的反向结果、幸存者偏差或测量变化；
3. 结论应是普遍规则、特定情境建议，还是下一轮实验；
4. 与最新官方规则、目标项目第一方证据是否冲突；
5. 修订后如何验证、何时停止或回退。

状态只允许：

- `pending-review`：等待人工判断；
- `approved-for-test`：可在明确范围内作为实验假设；
- `approved-for-reference`：用户明确批准有限定条件的方法论修订；
- `rejected`：记录理由，避免重复提案；
- `superseded`：被更新证据替代，保留历史链路。

即使批准进入 reference，也要保留来源观察、适用边界、反证、核验日期和停止条件。最新官方规则或当前项目 E2/E3 证据优先；历史学习不能覆盖政策事实。

## 六、矛盾与失效处理

- 方向相反的观察都保留，不删除不利结果；
- 先按市场、语言、站点阶段、页型、意图、实施质量和指标定义重新分层；
- 无法解释矛盾时，将假设保持待审或转为新的验证设计；
- 数据定义、平台报告或搜索功能变化时，标记受影响观察，不沿用旧结论；
- 后续高质量证据否定假设时，更新状态而不是覆写历史。

## 七、隐私与客户隔离

跨项目复用前执行数据最小化：

- 不记录公司名、域名、联系人、邮箱、电话、地址、账号 ID、客户名单、合同、报价或凭证；
- 不保留用户级查询与转化链路、原始收入/订单明细或可反推出客户的精确组合；
- 使用粗粒度行业/站点类型/市场标签和匿名编号，不把“化名”误当真正匿名；
- 指标只保留判断假设所需的聚合口径；小样本可能重新识别时进一步泛化或不写；
- 写入前对照当前项目已知标识符做检查，命中后删除或泛化；
- 不把客户一方提供的内部证据当成可向另一客户展示的案例；
- 用户要求查看、更正或删除时，应能定位对应记录和派生假设。

隐私检查失败的观察不得写入。不得为了证明假设收集超出当前授权范围的数据。

## 八、在项目中使用

已获授权读取学习库时，只检索与当前市场、页型、业务阶段和决策相关的条目，并在输出中明确标为“历史候选假设”。它可以帮助设计实验、提醒风险或调整预测区间，但不能替代当前项目的官方规则、第一方数据、SERP 样本和用户确认。

外部存储、搜索或分析能力均为可选。缺失时继续记录项目内 evidence/changelog；不创建未经授权的全局目录，不编造历史观察，也不因没有学习库而降低当前交付质量。

## 九、不要做

- 不把单例、相关性或顺利完成任务写成通用打法；
- 不用固定次数、固定周期或固定分数自动升级规则；
- 不让历史经验凌驾于官方规则和当前项目证据；
- 不把 AI 提及、内容格式、Schema 或第三方 SEO 指标当因果证明；
- 不自动修改 reference，不把笼统的“继续”视为方法论批准；
- 不跨客户保存或展示可识别信息。
