# 实战反馈回路 — 跨项目经验库（Learning Loop）

不复盘的方法论等于占星。本文件给 vibio 系列加上第二层记忆：`.vibio/` 是**项目记忆**（这个项目做到哪了），`~/.vibio-global/` 是**方法论记忆**（哪些打法被实战验证、哪些被证伪）。references 里的规则是出厂设置；经验库让它随每个真实项目的复盘越用越准。

核心机制一句话：**每次复盘把"会改变下次同类决策"的结论写进经验库；同一教训重复出现 ≥3 次，就升级为对 reference 本体的修订提案（人工确认后落地）。**

---

## 一、存储位置与结构

```
~/.vibio-global/
├── learnings.md       # 经验条目（append-only，按域分节）
└── calibration.md     # 校准队列：待人工确认的 reference 修订提案
```

放在用户主目录而非项目内——它要跨项目累积。首次写入时创建；文件不存在不报错，照常工作（skill 出厂状态就是没有经验库）。

**路径解析（执行细节）**：Read/Write/Edit/Grep 等文件工具需要字面绝对路径，不会展开 `~`。首次访问时先用 Bash 跑 `echo $HOME` 解析出真实路径（如 `/Users/xxx/.vibio-global/`），之后所有文件工具调用一律用解析后的字面路径——把 `~` 直接传给文件工具会得到"文件不存在"。

### learnings.md 条目格式

```md
## YYYY-MM-DD — <域> — <一句话结论>
- 域: keywords / content / links / geo / conversion / technical / recover
- 项目: <匿名代号，如 CF-厂A>
- 依据规则: <当时按哪条 reference 的哪个规则做的决策>
- 预期: <当时预期什么结果>
- 实际: <数据：排名/曝光/询盘/引用/链接，带时间窗口>
- 教训: <一句话，可操作，下次同类决策怎么改>
- 校准建议: none 或 <文件名 §节：建议怎么改>
- 置信度: 单例 / 重复出现(N=?) / 跨项目验证
```

### calibration.md 条目格式

```md
## <文件名> §<节> — <一句话修订提案>
- 支持证据: <引用 learnings.md 里的 ≥3 条同向条目日期>
- 现行规则: <原文>
- 提议改为: <新文>
- 状态: pending / approved / rejected / applied（YYYY-MM-DD）
```

## 二、回写协议（谁在什么时候写）

| 模式 | 时机 | 写什么 |
|---|---|---|
| **REVIEW** | 每次复盘收工时 | 对每个 Working / Not working 判定问一遍：这会改变下次同类决策吗？会 → 写条目。典型：某类验证 PASS 的词 8 周仍无曝光（Gate 判断疑似失准）；conditional 词出数据了（复核零量判定框架）；某内容形态在该行业拿到 AI 引用 |
| **RECOVER** | 每次恢复事件收尾 | 必写：原因诊断 + 真正起效的恢复动作（恢复案例天然稀缺，每个都值钱） |
| **LINK** | 季度安全体检时 | outreach 战术的 won/lost 汇总 vs playbook 预期转化率的偏差；哪类目录/平台真给了链接 |
| **WRITE** | 5 轮盲审仍未过 95 分时 | 失败案例的共性（哪个维度反复不过、什么原因） |
| **PLAN** | 6 个月重置触发时 | 哪个瓶颈诊断被证明是错的、真瓶颈是什么 |

**判据（与 changelog 的区别）**：changelog 记"做了什么"，经验库只记"学到了什么"。不改变未来决策的结果（如"按预期正常见效"）不写——经验库的价值密度靠克制维持。

## 三、读取协议（谁在什么时候读）

各模式开工读完项目 `.vibio/` 之后，**顺带读 `~/.vibio-global/learnings.md` 里自己域的小节**（用 Grep 按域过滤，不必全文读）。读到相关经验时在开工确认里带一句：「经验库：类似的 KD30+ 对比词在 2 个项目里 12 周才见曝光，本次预期按此校准。」

**经验与 reference 冲突时的裁决**：
- 置信度 = 单例 → reference 优先，经验仅作提醒
- 置信度 = 重复出现(≥3) 或跨项目验证 → 经验优先用于当前决策，并检查 calibration.md 是否已有对应提案，没有就补一条

**经验与经验矛盾时**（同域两条教训各自过阈值但结论相反）：都不进校准队列——给两条都加"存在矛盾观察"标记，并要求更细的域标签（如补上国家/行业/站龄维度）后重新分组计数。矛盾通常意味着分域太粗，不是其中一条错了。

## 四、校准机制（经验 → 方法论升级）

1. **触发**：写入新条目时检查同域同向教训是否已 ≥3 条（或 ≥2 个不同项目）。
2. **提案**：达到阈值就在 `calibration.md` 排队一条修订提案（格式见第一节），并在当次输出里告知用户：「经验库中『X』已第 3 次出现，建议修订 <文件> §<节>，已排入校准队列待确认。」
3. **确认**：**校准不自动改 reference。** 方法论本体的修改必须用户明确批准——批准的判定标准：用户消息中**点名了该提案**（文件名或提案标题）**且使用批准性措辞**（"批准"/"同意改"/"apply"）。笼统的"继续"/"可以"/"好"不构成批准，此时提案保持 pending 并明确追问。批准后按提案编辑 reference（含子技能副本同步），把提案状态改为 applied 并注日期。
4. **反向保护**：被 rejected 的提案留在 calibration.md（状态 rejected + 一句原因），防止同一提案反复被提起。

这与 skill 的其他硬闸门同构：盲审 95 分闸门管内容质量，校准双闸门（≥3 次重复 + 人工确认）管方法论质量。

## 五、隐私与客户边界

- 项目一律用匿名代号（行业+序号，如 `CF-厂A`），不写客户公司名、联系人、报价、单量。
- **写入前自检（强制）**：对照项目 `.vibio/project.md` 里的公司名/域名/联系人字符串检查条目文本，命中即替换为匿名代号再写入——匿名不能只靠自觉。
- 经验条目只保留做决策所需的抽象层（"KD30 对比词、新站、12 周见曝光"），不含可识别客户的细节。
- 经验库跨客户复用是它的价值所在，也正因此必须匿名——把这当成红线。

## 六、与现有模式的集成

| 模式 / 文件 | 接入点 |
|---|---|
| **REVIEW**（`review-engine.md`） | 复盘判定后新增"沉淀经验"步骤：按第二节判据写条目 + 检查校准触发 |
| **RECOVER**（`recovery-playbook.md`） | 恢复收尾必写经验条目 |
| **LINK** / **WRITE** / **PLAN** | 按第二节表格的时机回写；开工按第三节读取 |
| **KEYWORD**（`keyword-validation.md`） | conditional 词的 GSC 复核结果回流经验库——这是验证闸门自我校准的数据源 |
| `vibio-memory` | 项目记忆（第一层）与经验库（第二层）并立；读写工具同为 Read/Write/Edit，格式以本文件为准 |
| `predictive-seo.md` | 预测 vs 实际的调和结论是经验条目的天然来源 |

## 不要做

- 不记不改变决策的流水账——"按预期见效"不是经验。
- 单例观察不直接改 reference——先进经验库攒置信度。
- 校准提案不经用户确认不落地——方法论修改永远走双闸门。
- 不写客户可识别信息——匿名代号是红线。
- 不在经验库缺失时报错或强建——出厂状态就是空的，照常工作。
- 不把 changelog 内容复制进经验库——两者分工不同（做了什么 vs 学到了什么）。
