# Review Engine（闭环复盘）

AUDIT→FIX 之后总是断在「改完了，然后呢」。本引擎补上那一环：**改动 → 等足够时间 → 复测见效没 → 决定下一步**。它让 vibio 真正成为「运转中的操作系统」，而不是一次性咨询。

核心约束：**复盘要基于真实数据 + 改动历史，不靠感觉；见效要给搜索引擎留足重抓时间；结论引到官方指标。**

---

## 执行流程

**1. 读改动历史。** 用 `Read` 直接读项目根 `.vibio/`（格式约定见 `state-templates.md`）：`changelog.md`（改了什么、何时改、是否标记「面向 SERP 待重抓」）+ `project.md`（主瓶颈、阶段）+ `trackers/keywords.md`（上次排名快照）。没有 `.vibio/` 就说明没有可复盘的基线——引导先跑 AUDIT/FIX 建立记忆，或退化成一次普通审查。

**2. 判断重抓窗口够不够。** 面向 SERP 的改动（标题/描述、schema、canonical、内链、内容）通常要 **2-6 周** 才在 Google 稳定反映，新站更慢。对每条改动看「改动日期 → 今天」的间隔：
- 太近（< ~2 周）→ 老实说「还太早，数据噪声大」，给出建议复盘日期，**不要**把没变化判成「无效」。
- 够了 → 进入复测。
技术修复（修好 noindex、修 500）见效更快；纯展示类（OG 图）不影响排名只影响点击。

**3. 复测真实数据。** 路由专家，独立并行：
- `seo-google` — GSC 看每个目标页/词的**曝光、点击、CTR、平均排名**的前后变化；索引状态变化（之前没被索引的现在收录没）；GA4 看自然流量。这是见效与否的主证据。
- `seo-drift` — 对 SEO 关键元素做基线对比，确认改动**没有回归**（比如重新部署把修好的 canonical 又冲掉了），也抓出意外漂移。
- 必要时 `seo-dataforseo` 补独立的 SERP 排名核对。
数据不可得（没接 GSC/无 MCP）就如实说，退而比对可抓取的渲染信号 + 用户能提供的截图。

**4. 逐项判定见效。** 把每条改动对到它的预期指标，按以下定量阈值判结论：

| 结论 | 判定条件 | 下一步 |
|------|---------|--------|
| **见效** | 满足任一：(a) 目标页面 avg position 改善 ≥3 位且维持 ≥2 周，(b) CTR 相对同位置预期值提升 ≥15%，(c) organic clicks 月环比 +20%，(d) 新收录页面 ≥90% 进入索引且开始获得曝光 | 收尾，资源挪向下一个瓶颈 |
| **未见效** | 重抓窗口已过（≥6 周），以上条件无一满足 | 根因分析 → 刷新/加深度/补内链 或 合并/重写/降级 |
| **回归** | seo-drift 检测到关键元素（canonical/title/schema/noindex）发生非预期变更 | 标记重修（转 FIX） |
| **太早** | 距改动 <4 周（新站 <6 周），或 GSC 数据尚未包含重抓后的稳定信号 | 定下次复盘日期 |

**CTR 基准：** 使用 `predictive-seo.md` §3 的 Position-CTR 查表。position-adjusted CTR < 预期值的 70% → 判定为 CTR 偏低。

**统计显著性快速判定：** impressions ≥500/月 → 可检测 ±20% CTR 变化；≥1000/月 → 可检测 ±10% 变化。更精确走 `seo-experimentation.md` §7 的两比例 z-test。

**5. 决定下一步。** 不只报数字，要给动作：
- 排名卡在 11-20（第二页）→ 刷新内容/加深度/补内链（`seo-content` + 转 FIX）。CTR < 预期值 70% → 优先改 title/description。

**5. 决定下一步。** 不只报数字，要给动作：
- 排名卡在 11-20（第二页）→ 刷新内容/加深度/补内链（`seo-content` + 转 FIX）。
**6. 写回记忆。** 用 `Edit`/`Write` 直接操作 `.vibio/`：更新 `trackers/keywords.md` 的当前排名 + 趋势列；把本次复盘结论追加到 `changelog.md`（带日期、Type: REVIEW、每项见效判定、下一步）；必要时更新 `project.md` 的阶段/主瓶颈（一个瓶颈解决了就推进到下一个）。

---

## 节奏

对齐 operating-system.md 的复盘节奏：**月度**深度复盘是主循环；单次重大修复后 2-6 周做一次定向复测。不做日常焦虑型排名检查——每天看排名只会看到噪声。

---

## 下一步路由（复盘完成后）

| 触发条件 | 推荐 |
|---------|------|
| 某项改动被回归冲掉，或需补强（CTR 低改标题、第二页加深度）| 复盘发现这些项要再动手：转 FIX 模式，改完照例写回 changelog。 |
| 一个瓶颈已解决，该推进路线图下一阶段 | 这个瓶颈见效稳定了。转 PLAN 模式把重心挪到下一个瓶颈、更新路线图。 |
| 数据暴露出新的、之前没诊断的问题 | 复盘带出一个新问题（如某批页突然掉收录）。转 AUDIT 定位根因。 |
| 窗口还不够 | 现在复盘太早，搜索引擎还没充分重抓。建议 <日期> 再来，我已把待复盘项记进 changelog。 |

---

## 不要做

- 不在重抓窗口不够时就把改动判成「无效」——那是噪声，不是结果。
- 不靠感觉/印象下结论，复盘必须基于 GSC/GA4 真实数据或可抓取信号。
- 不只报数字不给动作——复盘的产出是「下一步做什么」。
- 不承诺「再等等一定会涨」——诚实区分「时间不够」与「方法没用，需换路子」。
- 不跳过 drift 检查就假设改动还在生效（重新部署常把修复冲掉）。
- 复盘完不写回记忆——那等于又断了闭环。
